# Domény a systém doménových jmen (DNS - Domain Name System)

[DNS](https://grokipedia.com/page/Domain_Name_System) je
hierarchický, distribuovaný systém, který překládá názvy domén
(a dalších prvků) na IP adresy. (ARPANET, 1983)

Nejnižší úroveň (původní řešení: hosts.txt): `/etc/hosts`

Stromová hierarchie, hierarchický jmenný prostor, rozdělený do
zón.

1. kořenové servery: obsluhují dotazy na top-level domény
   (prvního řádu / úrovně)
1. TLD servery obsluhují dotazy na domény druhého řádu (second
   level)
1. nižší servery dotazy na subdomény

Redundance, kešování TTL (time-to-live), dotazy přes UDP nebo
TCP, port 53, delegace autority pro nižší úrovně.
Autoritativní odpovědi kryptograficky podepsané.

## TLDs (Top-Level Domains)

- origins in ARPANET Era (1970s-1980s)
- spravuje [ICANN](icann.org)

### Obecné (generic) TLDs

Dnes > 1500, původních "klasických sedm":

- `.arpa` - infrastructure
- `.com` - komerční entity, nejpočetnější
- `.edu` - U.S. vzdělávácí instituce
- `.gov` - U.S. vládní instituce
- `.mil` - U.S. armáda
- `.org` - neziskové organizace
- `.net` - network infrastructure providers
- `.int` - mezinárodní organizace

2026 další nové TLDs.

### Kódy zemí a oblastí (country-codes, territories) TLDs

První: .us, .uk, .ca, .fr, .de

## Hierarchical Name Space

Identifikátory oddělěné `.` čteme zprava doleva,
case-nonsensitive. Nepojmenovaný (null) kořen `.`

Celé jméno domény: délka < 255, dnes Unicode.

## Name Servers (NS), zóny

Autority spravují zóny, Authoritativní NS spravují veškeré
aktuální záznamy pro své zóny. Pro každou zónu existují
alespoň dva NS pro redundanci. NS vyšších úrovní neukládají
data pro podzóny, ale autoritu delegují.

- [ICANN](https://www.icann.org/en/beginners)
- [IANA](https://cs.wikipedia.org/wiki/IANA)

[Kořenový serverový systém](https://cs.wikipedia.org/wiki/Domain_Name_System#Root_servery)
má 13 NS, spravovaný 12ti organizacemi (celkem ~2000
instancí), delegují správu na TLD autoritativní servery.

## Address Resolution Process

Prohlížeč - jméno domény > local resolver (DNS client) >
přepošle resolveru ISP (internet service provider) nebo
otevřenému NS, např.

- `8.8.8.8` (Google)
- `1.1.1.1` (Cloudflare)
- `tracepath 8.8.8.8`

Hledá se v místní keši (local cache, přítomnost záznamu, TTL
time-to-live), podle potřeby nahoru stromem resolverů.

## Typy záznamů v NS

- A (IPv4)
- AAAA (IPv6)
- CNAME (canonical name)
- MX (mail exchange)
- NS (name server)
- TXT (text)
- PTR (pointer, reverse lookup)

Nástroje:

- `nslookup` (deprecated)
- `host` (simple)
- `dig` (DNS lookup utility)

```bash
dig ujep.cz # SERVER: 127.0.0.53#53  A record
dig @8.8.8.8 ujep.cz
dig ujep.cz A # A AAAA MX NS TXT CNAME
dig ujep.cz CNAME # canonical name
dig ujep.cz NS # name servers
dig ujep.cz SOA # start of authority
dig -x 195.113.141.132 # reverse lookup
```

```bash
dig rqa.cz
dig -x 95.111.226.156
```

```bash
dig +trace ujep.cz
dig +trace pm.me
```

```bash
host ujep.cz
host -t A ujep.cz
```

```bash
dig +short ujep.cz
dig -x $(dig +short ujep.cz)
```

# SSL certifikáty

Nutné pro zabezpečení (šifrování) komunikace (HTTPS) a
důvěryhodnost domény (je to ta, za kterou se vydává) - podpis
certifikátu důvěryhodnou certifikační autoritou (CA),
zobrazení bezpečnostního indikátoru (šedivý/zelený zámek v
prohlížeči).

Certifikační autority mají své kořenové certifikáty, které
jsou předinstalované v prohlížečích a operačních systémech.

Kořenové certifikáty (Root Certificates) jsou důvěryhodné
(Microsoft/Apple/Mozilla/Google), další certifikátory
podepsané těmito autoritami jsou také důvěryhodné.

Státy a organizace mívají své vlastní certifikační autority,
certifikáta jsou důvěryhodné pro specifické účely.

## Certifikát pro doménu

Řetězec důvěry: certifikát pro doménu je podepsán certifikační
autoritou, která je důvěryhodná (její certifikát je podepsaný
autoritou,jejíž kořenový certifikát je předinstalován v
prohlížečích).

Ověření: server pošle certifikát pro doménu, prohlížeč
zkontroluje podpis a řetězec důvěry, zobrazí indikátoru
zabezpečení.

Při selhání ověření:

- zobrazení varování: "Neplatný certifikát", "Neznámý
  vydavatel", "Nesoulad domény", apod.

## Process získání certifikátu

ACME (Automatic Certificate Management Environment): klient si
vyžádá certifikát od CA, CA ověří vlastnictví domény (např.
pomocí HTTP-01 challenge, DNS-01 challenge, TLS-ALPN-01
challenge), a vydá certifikát.

- ACME klient
  - vytvoří klíčový pár (private key, public key)
  - vytvoří žádost o certifikát (CSR - Certificate Signing
    Request) podepsaný privátním klíčem, obsahující veřejný
    klíč a informace o doméně
  - odešle žádost CA

- CA
  - ověří vlastnictví domény pomocí challenge
  - vytvoří certifikát pro doménu, podepíše ho svým privátním
    (intermediate) klíčem
  - odešle certifikát klientovi

- Klient
  - obdrží certifikát a uloží ho
  - nakonfiguruje webový server pro jeho použití

Certifikáty mají omezenou platnost (např. 90 dní pro Let's
Encrypt), po které je nutné je obnovit. ACME klient může
automaticky v4as obnovovat certifikáty.

### Challenge pro ověření vlastnictví domény

- HTTP-01: klient vytvoří specifický soubor s tokenem na svém
  webovém serveru, CA se pokusí tento soubor načíst přes HTTP
  a ověří, že klient má kontrolu nad doménou.
- DNS-01: klient vytvoří specifický TXT záznam v DNS pro svou
  doménu, CA ověří tento záznam a tím potvrdí vlastnictví
  domény.

## Let's Encrypt

[Let's Encrypt](https://letsencrypt.org/) je důvěryhodná
certifikační autorita, která poskytuje bezplatné SSL
certifikáty pro zabezpečení webových stránek.

Nainstalovaný kořenový certifikát (ISRG Root X1):

- prohlížeče přebírají od operačního systému.
- kořenový certifikát je self-signed (issuer a subject jsou
  stejné).

Kde najít kořenové certifikáty (root store - nutno chránit!):

- Windows: certmgr.msc
- Linux: /etc/ssl/certs/ca-certificates.crt (vygenerovan7o z
  jednotlivých .pem souborů v /etc/ssl/certs/)
- macOS: Keychain Access
- Firefox: about:certificate, Authorities (má vlastní
  úložiště)

PEM (Privacy-Enhanced Mail, zastaralé jméno) je formát pro
certifikáty.

````
Linux:

```bash
cd /etc/ssl/certs
ls
cat ca-certificates.crt
cat ISRG_Root_X1.pem
openssl x509 -in ISRG_Root_X1.pem -text -noout
````

## Certbot

Certbot je ACME klient pro Let's Encrypt.

```bash
# as root:
apt install certbot python3-certbot-apache
certbot --apache certonly -d rqa.cz -d www.rqa.cz
```

#### privkey.pem (private key, tajný klíč)

Soukromý klíč serveru, nikdy se nesdílí, používá se pro

- digitální podpis při navazování TLS spojení (handshake)
- tedy důkaz, že vlastním certifikát
- ve starších verzích (dnes již ne): dešifrování dat
  šifrovaných veřejným klíčem

#### fullchain.pem (veřejný klíč + intermediate certifikát)

Obsahuje:

- serverový certifikát (např. rqa.cz), který mj. obsahuje
  veřejný klíč
- intermediate certifikát(y): „důvěryhodný řetězec
  certifikátů“
